**Short answer.** No — it stewards, which is different in a specific, checkable way. The Association writes and maintains the node software and standards, but the network is run by independent operators who choose whether to adopt any release. Maintaining the software that others voluntarily run is influence through usefulness, not control through authority.

## More detail

The confusion is understandable: the Association is the most visible organization in the ecosystem, it maintains the reference software ([[how-is-bsv-developed]]), and it speaks publicly for the technology. Visibility reads as control. But test it against what control would mean:

- Can it reverse a transaction, freeze coins, or censor an address? No — nobody can; that's the design ([[how-do-transactions-work]], [[what-bsva-does-not-do]]).
- Can it force operators to run a software change? No — operators are independent businesses that adopt what serves them ([[who-runs-the-bsv-network]]); a rejected release simply doesn't run. Blockchain history's forks, including BSV's own origin ([[how-did-bsv-start]]), are what operator disagreement looks like in practice.
- Can it set fees or decide listings? No — fees emerge from the market ([[what-are-fees]]); listings are exchanges' decisions ([[what-is-a-listing]]).

What it *can* do is real but bounded: shape the software roadmap, steward standards, fund education, promote adoption ([[what-is-the-bsv-association]]). BSV's "set in stone" protocol policy narrows even the roadmap lever — the stated commitment is to keep the base rules stable rather than steer them.

Why the distinction matters beyond pedantry: attributing control to the Association leads to blaming it for things it cannot decide ([[orange-gateway-mica]], [[exchange-problems-mean-bsv-problems]]) — and to missing who actually made a given decision ([[ecosystem-who-is-who]]).
